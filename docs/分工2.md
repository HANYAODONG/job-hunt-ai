# 第二阶段工作流分工

## 1. 本阶段目标

本阶段不继续扩充爬虫，也不立即训练大模型。先围绕现有大公司岗位、合成简历和标准岗位词典，完成一条可评测、可替换、可解释的人岗推荐链路：

```text
岗位与简历标准化
  -> BM25 召回 Top200
  -> text2vec 语义重排
  -> 技能抽取与标准化
  -> 知识图谱特征
  -> 多因子融合排序
  -> 金标评测与推荐解释
```

当前统一输入：

- `dataset/incoming/job_bigcompany_final.csv`
- `dataset/incoming/synthetic_detailed_resumes.csv`
- `dataset/incoming/standard_job_title_dictionary.csv`

其中 `standard_job_title_dictionary.csv` 只负责岗位名称标准化，不能代替标准技能词典。

## 2. 工作流 1：数据基座、LLM 标注与评测

负责人：韩耀栋

### 主要任务

1. 维护岗位、简历和标签的统一字段契约。
2. 生成标准 `jobs.jsonl` 和 `candidate_profiles.jsonl`。
3. 用 BM25 为每份简历生成候选岗位，避免构造“简历数 x 全部岗位数”的笛卡尔积。
4. 调用 LLM 判断简历与岗位的匹配等级，并要求返回判断证据。
5. 对 LLM 标签做一致性检查、抽样人工复核和版本管理。
6. 维护 Recall、MRR、NDCG 等评测脚本和基线报告。

### LLM 标注方案

每份简历建议选择 15 至 25 个岗位对：

- BM25 前排岗位：构造可能的正样本。
- BM25 中间位次岗位：构造困难正样本或困难负样本。
- 不同岗位族的随机岗位：构造明显负样本。

例如 200 份简历、每份 20 对，只需判断约 4000 对，而不是判断 200 x 10000 个组合。

LLM 对每个数据对输出：

```json
{
  "candidate_id": "resume_001",
  "job_id": "job_001",
  "grade": 2,
  "hard_constraint_pass": true,
  "matched_skills": ["Python", "SQL"],
  "missing_required_skills": ["PyTorch"],
  "resume_evidence": ["熟悉 Python 数据分析"],
  "job_evidence": ["要求掌握 Python 和深度学习框架"],
  "confidence": 0.86,
  "label_source": "llm"
}
```

等级约定：`0` 不相关、`1` 弱相关、`2` 较相关、`3` 强相关。

LLM 直接生成的结果先称为“候选金标”或“高质量银标”。建议随机抽查 10% 至 20%，并重点复核低置信度、两次判断不一致和边界样本。经过人工确认的数据才能进入正式测试金标，避免用模型自己的偏好评测模型。

## 3. 工作流 2：text2vec 轻量语义重排

负责人：叶骑瑞

### 技术定位

`text2vec` 可以作为本阶段的轻量语义模型方案。它通常比 BGE-M3 占用更少的显存和内存，部署更简单，适合先跑通中文人岗匹配流程；但“效果差不多”不能预先假定，必须在同一金标集上实验比较。

本阶段建议：

- text2vec 作为默认轻量 baseline。
- BGE-M3 作为效果对照组，不作为本地开发前置条件。
- 对 BM25 Top200 直接计算余弦相似度即可，暂时不需要 HNSW。
- 岗位向量离线生成并缓存，简历向量在查询时生成。

### 主要任务

1. 确定 text2vec 中文模型、文本拼接方式和最大长度。
2. 离线生成岗位向量及岗位 ID 映射。
3. 读取工作流 5 的 BM25 Top200 候选集。
4. 计算简历与候选岗位的余弦相似度并重新排序。
5. 输出 `semantic_score` 和 `semantic_rank`。
6. 对比 BM25、BM25 + text2vec、BM25 + BGE-M3 的效果和资源占用。

建议分别测试以下岗位文本：

- 仅岗位标题。
- 标题 + 岗位要求。
- 标题 + 职责 + 要求 + 标准技能。

### 统一输出

```json
{
  "query_id": "resume_001",
  "candidates": [
    {
      "job_id": "job_001",
      "semantic_score": 0.82,
      "semantic_rank": 1
    }
  ]
}
```

### 验收标准

- 模型能在普通本地环境运行。
- 岗位向量可重复生成并缓存。
- 能完成 BM25 Top200 语义重排。
- 报告 Recall@K、MRR、NDCG@K、编码耗时、模型大小和峰值内存。
- 根据评测结果决定保留 text2vec，还是正式实验切换至 BGE-M3。

## 4. 工作流 3：技能抽取、标准化与知识图谱

负责人：魏昊朗

### 第一部分：建立标准技能词典

新建独立的 `standard_skill_dictionary.csv`，不要把岗位词典当成技能词典。建议字段：

```csv
skill_id,canonical_name,aliases,skill_category,parent_skill,match_pattern,source,version
```

示例：

```csv
SK001,提示词工程,"Prompt;Prompt Engineering;提示词",大模型应用,生成式AI,,team_review,v1
SK002,检索增强生成,"RAG;Retrieval-Augmented Generation",大模型应用,生成式AI,,team_review,v1
SK003,智能体开发,"Agent;AI Agent;Agent开发",大模型应用,生成式AI,,team_review,v1
```

`standard_job_title_dictionary.csv` 继续负责：

```text
原始岗位标题 -> 标准岗位 -> 岗位类别/岗位族
```

`standard_skill_dictionary.csv` 负责：

```text
原始技能表达 -> 标准技能 -> 技能类别/上位技能
```

### 第二部分：技能抽取与标准化

1. 从岗位的 `skills`、`traditional_skills`、`new_skills` 字段读取已有技能。
2. 从 `job_requirement`、`job_responsibility` 和简历文本中补充抽取技能。
3. 先用词典、别名和规则完成高精度匹配。
4. 对未登录词使用 LLM 或语义相似度生成候选映射，但必须保留置信度和人工审核状态。
5. 区分必备技能、加分技能、通用能力和行业知识。
6. 输出原始词、标准词、文本证据、来源字段和置信度。

### 第三部分：知识图谱与匹配特征

图谱最小节点：`Job`、`JobFamily`、`Skill`、`Candidate`、`Company`、`Time`。

图谱最小关系：

- `Job-[:BELONGS_TO]->JobFamily`
- `Job-[:REQUIRES_SKILL]->Skill`
- `Candidate-[:HAS_SKILL]->Skill`
- `Skill-[:RELATED_TO]->Skill`
- `Skill-[:PREREQUISITE_OF]->Skill`

工作流 3 不只负责画图，还要输出推荐算法可以使用的特征：

```json
{
  "query_id": "resume_001",
  "job_id": "job_001",
  "matched_skills": ["Python", "SQL"],
  "missing_skills": ["PyTorch"],
  "skill_coverage": 0.67,
  "job_family_match": 1.0,
  "graph_relatedness": 0.72,
  "evidence_paths": []
}
```

### 交付物

- `standard_skill_dictionary.csv` 第一版
- 技能抽取与标准化脚本
- 技能词典覆盖率、未登录词和人工抽查报告
- Neo4j schema 与可重复导入脚本
- 能力差距和图谱相关度特征文件

## 5. 工作流 4：融合排序、解释与前端

负责人：纪雨涵

### 主要任务

1. 合并 BM25、text2vec、技能覆盖、岗位族和图谱特征。
2. 对不同来源的分数进行归一化。
3. 先实现可解释的加权融合，再根据金标尝试学习权重。
4. 展示最终排名、分项得分、匹配技能、缺失技能和证据。
5. Mock 页面只用于并行开发，真实接口完成后必须替换模拟数据。

第一版可采用：

```text
final_score =
  w1 * bm25_score
  + w2 * semantic_score
  + w3 * skill_coverage
  + w4 * job_family_match
  + w5 * graph_relatedness
```

### 验收标准

- 能读取其他工作流的统一输出。
- 能输出 TopN 推荐、分数明细和推荐解释。
- 与单独 BM25、单独 text2vec 做消融对比。
- 前端结果不再依赖固定 Mock 数据。

2. 导入 Elasticsearch 并维护可重复构建的索引脚本。

### 验收标准

## 7. 工作流依赖关系

```text
工作流 1：标准数据与标签
  -> 工作流 5：BM25 候选集
  -> 工作流 2：text2vec 语义重排
  -> 工作流 3：技能与图谱特征
  -> 工作流 4：融合排序与展示

工作流 5 的候选集
  -> 回流工作流 1，供 LLM 选择待标注数据对
```

为避免互相等待：

- 工作流 2 可先使用工作流 5 已有候选集或固定小样例。
- 工作流 3 可先对 100 条岗位和 20 份简历建立词典与小图。
- 工作流 4 可先使用统一格式的 Mock 输入开发，但最终验收必须接真实结果。
- 所有人不得自行改变公共 ID 和字段名；需要修改时先更新 `docs/data-schema.md`。

## 8. 本轮统一验收

本轮结束时至少完成一次可复现实验：

```text
标准简历
  -> BM25 Top200
  -> text2vec 重排
  -> 技能覆盖与图谱特征
  -> 融合 Top10
  -> 对照正式金标计算指标
  -> 前端展示推荐原因与能力缺口
```

必须保留实验配置、模型名称、数据版本、随机种子、运行时间和指标结果。最终报告至少比较：

- BM25
- BM25 + text2vec
- BM25 + text2vec + 技能特征
- BM25 + text2vec + 技能特征 + 图谱特征
